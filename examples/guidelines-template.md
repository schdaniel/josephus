## Audience

<!-- Describe who will read this documentation -->
<!-- Examples: developers, end-users, system administrators, etc. -->

Technical developers integrating with our API.

## Tone

<!-- Describe the writing style -->
<!-- Examples: formal/informal, technical/approachable, concise/detailed -->

Technical but approachable. Use clear language and avoid jargon where possible.

## Scope

### Include

<!-- What should be documented -->

- All public APIs and their parameters
- Installation and setup instructions
- Common use cases and examples
- Error handling and troubleshooting

### Exclude

<!-- What should NOT be documented -->

- Internal implementation details
- Deprecated features
- Admin-only functionality

## Structure

<!-- Preferred documentation organization -->

- Start with a quick-start guide
- Group related features together
- Include a comprehensive API reference
- Add a troubleshooting/FAQ section

## Style

### Code Examples

<!-- Preferred languages and styles for code examples -->

- Use Python for backend examples
- Use JavaScript/TypeScript for frontend examples
- Include complete, runnable examples when possible

### Formatting

<!-- Preferred formatting conventions -->

- Use tables for parameter documentation
- Include request/response examples for APIs
- Use Mermaid diagrams for workflows
